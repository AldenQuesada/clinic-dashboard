1 Visão geral e onde a Lara roda

Erros conhecidos a não repetir:

  • Regex com \n em JSON quebra — usar split(String.fromCharCode(10))
  • $json após HTTP node retorna response body, não os dados originais — usar $("NodeName").first().json
  • sleep(4000) fixo é frágil — pipeline sequencial elimina necessidade
  • Concatenar follow-up no texto principal é errado — logar como msg separada via p_follow_up

7 Funcionalidades

Cadência (follow-ups)

Steps em wa_cadences, disparados por wa_run_cadences (pg_cron 30min, 8-20h BRT).

Full Face: Day 1 (IMPLICATION+foto) → Day 2 (VALOR) → Day 3 (ESCASSEZ) → Day 5 (PUXÃO) → Day 7 (PORTA ABERTA) → Day 10 (ENCERRAMENTO).

Procedimentos: Day 1-5 versão curta.

Só ativa se lead tem funil setado.

Nudge (reativação)

Máximo 1 por conversa (flag reactivation_sent, resetada em nova msg do lead).

Só horário comercial: seg-sex 8-19h, sáb 8-13h, dom OFF.

Reativação automática da secretária: 2min sem resposta e Lara retoma (wa_auto_reactivate).

Broadcast

Tabela wa_outbox + workflow dedicado (x5f4MApEcSUX5hca, processor 2min).

Anti-bloqueio batch=1, substituição [nome], throttle configurável.

Fotos antes/depois

Tag [FOTO:queixa] no output da Claude. Queixas: geral, olheiras, sulcos, flacidez, contorno, papada, textura, rugas, rejuvenescimento.

~16 fotos em wa_media. 2 por envio, pessoas diferentes, exclui já enviadas.

PIX automático

CNPJ 55.167.164/0001-08 — M CLINIC ESTHETIC EXPERT LTDA.

Lara pergunta sobre o que é o pagamento, pede comprovante.

Detecção de funil

wa_detect_funnel_from_message na primeira msg. Trigger-based (não classificador).

Sem funil → Lara atende genérico ou é bloqueada pelo guard no_funnel (depende da fase).

Modo híbrido

Secretária assume via Inbox: ai_enabled=false + tag human_assumed.

Lara reativa se secretária parar 2min (só se paused_by NÃO for manual/human_handoff/inappropriate/auto_limit).

Blacklist

wa_phone_blacklist — staff, fornecedores, testes. Seed com 21 telefones.

wa_log_message retorna {success:false, blacklisted:true} sem criar nada.

Madrugada

Lara responde se lead mandar msg (24/7).

Lara não envia nudge/reativação fora do horário comercial.

Agendamento (handoff)

Lara NUNCA oferece horários. Pergunta: manhã/tarde + dia preferencial. Passa pra equipe.

Outbox via wa_outbox_schedule_automation dispara confirmação + lembretes.

13 Bugs operacionais reportados (críticos)

A. Lara não respeita o botão "Parar" Crítico

Sintoma: secretária clica "Assumir"/pausar a conversa mas a Lara continua enviando mensagens (especialmente cadência e broadcast).

Como o botão funciona (OK):

  • js/ui/inbox.ui.js:446 aciona InboxService.assume()
  • js/repositories/inbox.repository.js:54 chama RPC wa_inbox_assume
  • RPC seta ai_enabled=false e tag human_assumed em wa_conversations
  • O Guard Check do workflow n8n respeita: wa_guard_check retorna action='block' quando ai_enabled=false e o workflow desvia para "Log Inbound Only"

Onde quebra (causa raiz):

ComponenteArquivoProblema
Cadência (wa_run_cadences) supabase/migrations/20260597000000_cadence_engine.sql Não verifica ai_enabled antes de inserir na wa_outbox. Agenda follow-ups mesmo pausada.
Outbox processor n8n/outbox-processor-workflow.json Busca wa_outbox_fetch_pending apenas por status='pending'. Não cruza com wa_conversations.ai_enabled antes de enviar.
Broadcast Mesma engine do outbox Ignora totalmente o estado da conversa. Se você agendou broadcast e depois pausou a conversa, ela recebe assim mesmo.
Race condition Workflow n8n Se a secretária clica pausar enquanto a msg já entrou no branch "allow", a resposta sai antes do update propagar. Sem lock pessimista.

Correção proposta:

  1. Em wa_run_cadences: adicionar AND c.ai_enabled = true na query que insere na outbox.
  2. Em wa_outbox_fetch_pending: JOIN com wa_conversations e filtrar ai_enabled = true OR sender_override = 'humano' (permitir broadcast manual).
  3. Na pausa manual: marcar outbox pendente como status='cancelled' para essa conversa (cleanup ativo).
  4. Race: adicionar re-check de ai_enabled imediatamente antes do Send WhatsApp no workflow (custo: +1 round-trip Supabase; benefício: elimina race).

B. Mensagens não chegam no Inbox Alta

Sintoma: lead manda msg pelo WhatsApp, mas não aparece no Central de Atendimento.

Filtros em Parse Message (n8n/lara-whatsapp-workflow.json):

  • fromMe = true sem texto → descartado (linha ~47)
  • Grupo (@g.us) e broadcast (@broadcast) → descartado (linha ~64)
  • source = 'api' em send.message → descartado (é a própria Lara respondendo, evita loop)
  • source permitidos implicitamente: android, web, ios. Valores como desktop, unknown, ou novos da Evolution podem ser filtrados sem intenção.

Falhas silenciosas:

  • wa_log_message (migration 20260627_inbox_consolidated_v2) não grava em wa_errors se algo dá errado na busca de conversa. Msg pode ser "aceita" pelo webhook mas nunca aparecer.
  • Se o telefone está em wa_phone_blacklist, retorna {success:false, blacklisted:true} silencioso — correto comportamento, mas não há UI que mostre "esta mensagem foi bloqueada por blacklist".

Correção proposta:

  1. Logar em wa_errors toda execução de wa_log_message que não criou mensagem (com error_type='silent_drop').
  2. Adicionar desktop ao whitelist de source (WhatsApp Desktop é comum entre secretárias).
  3. Dashboard: painel simples lendo wa_errors das últimas 24h.

C. Conversas duplicadas Crítico

Sintoma: mesmo telefone aparece com 2+ conversas ativas simultâneas.

Causa raiz:

  • Não existe unique index em wa_conversations na migration base 20260582_messaging_system. Apenas índices comuns (idx_wa_conversations_lead, idx_wa_conversations_status).
  • A memória indicava um unique index parcial (clinic_id, phone) WHERE status='active' — verificação atual não encontrou essa constraint. Possivelmente foi removida ou a migration nunca foi consolidada.
  • wa_find_conversation faz 3 matches em cascata: phone exato → remote_jid LIKE %jid% → right(phone,8) fuzzy. O fuzzy pode achar a "errada" e então o insert que fica perdido pode criar nova em paralelo.
  • Race: 2 mensagens chegando em ms de intervalo → duas transações rodando wa_find_conversation ao mesmo tempo, ambas retornam NULL, ambas criam conversa.

Correção proposta:

CREATE UNIQUE INDEX CONCURRENTLY ux_wa_conversations_phone_active
  ON wa_conversations (clinic_id, phone)
  WHERE status = 'active';

-- Em wa_log_message, envolver find+create com:
INSERT INTO wa_conversations (...)
VALUES (...)
ON CONFLICT (clinic_id, phone) WHERE status='active'
DO UPDATE SET last_inbound_at = now()
RETURNING id;

Isso elimina race e força 1 conversa ativa por telefone. Migrar duplicatas existentes: merge mensagens na conversa mais antiga, marcar as outras status='merged'.

D. Nem todas as conversas aparecem no Inbox Média

Sintoma: conversas existem no banco mas somem da lista do Inbox.

Onde filtra:

  • wa_inbox_list (migration 20260591): WHERE status='active' — conversas archived/closed/merged não aparecem.
  • Sem paginação/limit na RPC, mas o frontend mantém _conversations em cache (js/ui/inbox.ui.js). Se o realtime subscription falhar, não refresha.
  • Filtros de UI (inbox.ui.js:270-295): "waiting" mostra !ai_enabled, "lara" mostra ai_enabled. Uma conversa resolvida pela secretária com status='archived' some de qualquer filtro exceto "resolved".

Gaps identificados:

  • Sem botão "ver arquivadas" proeminente — secretária pensa que "sumiu".
  • Realtime (Supabase Realtime) pode estar caindo sem reconnect automático, deixando lista defasada até F5.
  • Conversas com remote_jid quebrado (LID format) aparecem "Desconhecido" — hierarquia: lead.name → display_name → phone formatado. Se todos falham, some visualmente.

Correção proposta:

  1. Indicador visual de conexão realtime (bolinha verde/vermelha no topo do Inbox).
  2. Botão "mostrar arquivadas" no filtro.
  3. Fallback: polling a cada 30s se realtime cair.
  4. Para "Desconhecido": forçar display_name = phone formatado sempre, nunca deixar null.

E. Mensagens invertidas no histórico Média

Sintoma: resposta da Lara aparece ANTES da pergunta do lead no chat.

Causa raiz (histórica): inbound e outbound com mesmo sent_at. Corrigido via wa_log_message_sequential com timestamps incrementais (inbound now(), outbound +1s, fotos +2s/+3s, follow-up +4s).

Alerta: se alguém reintroduzir conexões paralelas no workflow n8n após Process AI Response, o bug volta. Regra inegociável: pipeline sequencial.

F. Persona não avança Alta

Sintoma: Lara fica no tom "onboarder" mesmo quando o lead já está pronto pra fechar.

Causa: wa_log_message_sequential seta ai_persona na primeira mensagem e nunca mais atualiza. Não há trigger de promoção baseada em N mensagens ou tags [PRONTO_AGENDAR]/[PERGUNTOU_PRECO].

Correção: após cada mensagem processada, rodar regra simples:

  • msg_count ≥ 4 + temperature ≥ warm → promover para sdr
  • msg_count ≥ 8 + tag perguntou_preco → promover para closer
  • fase = scheduled → confirmador
  • fase = no_show → recuperador
