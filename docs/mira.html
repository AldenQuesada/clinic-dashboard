Conceitos fundamentais

SSOT
Single Source of Truth. Toda a lógica Tier 1 vive em wa_pro_handle_message. Dashboard e n8n apenas chamam essa RPC. Zero duplicação.
Tier 1
Regex puro em PL/pgSQL. Classifica a mensagem em 1 de 12 intents. ~0ms de latência. Cobre 90% dos casos comuns.
Tier 2
Claude Haiku (claude-haiku-4-5) com tool use. Acionado quando Tier 1 retorna unknown. Decide qual tool chamar. Custo ~R$5-10/mês.
Access Scope
own / team / full. Determina se o profissional vê só seus dados, do time ou da clínica toda.
Two Worlds
Números oficiais da clínica → Lara (atende pacientes). Números privados dos profissionais → Mira (consultas internas). Cross-talk impossível.

SSOT — Single Source of Truth

Antes da refatoração, a mesma lógica Tier 1 existia em 2 lugares:

  1. js/services/mira.service.js (dashboard console)
  2. Code node "Mira Brain" dentro do workflow n8n (115 linhas JS inline)

Qualquer correção (regex, formatter, nova intent) precisava ser feita nos 2 lugares. Um bug no regex patient_balance foi corrigido no dashboard mas ficou esquecido no n8n durante 1 sessão.

Solução
Toda a lógica foi movida pra uma RPC única wa_pro_handle_message(phone, text) no Postgres. Dashboard e n8n agora apenas chamam essa RPC. Fonte única de verdade, zero drift.

🩺 Pré-Consulta — 10min antes

Cron a cada 5min verifica se há consultas começando em ~10min (janela 8-13min). Alerta o profissional antes da pessoa entrar no consultório.

Janela deslizante

-- Só alerta se appointment começa entre 8 e 13 min a partir de agora
WHERE (v_today + a.start_time::interval) AT TIME ZONE 'America/Sao_Paulo'
      BETWEEN now() + interval '8 minutes' AND now() + interval '13 minutes'

Como roda a cada 5min, cada appointment dispara o alerta exatamente 1 vez (a janela de 5min garante isso).

Formato

⏰ *Patricia* chega em ~10min
─────────────
Horario: 14:00
Procedimento: Lifting 5D
Valor: R$ 600,00
Impacto
"Mirian nunca mais entra numa consulta sem contexto." — Antes de cada atendimento, ela sabe quem é, o que vai fazer, quanto vai cobrar.

🎯 Anomaly Detection — 22h

Todo dia às 22h, compara faturamento do dia vs média + stddev dos últimos 30 dias. Se desvio > 2σ (top/bottom 2.5%), envia alerta.

Algoritmo

delta_sigma = ABS(today - avg_30d) / stddev_30d

IF delta_sigma > 2 AND today > 0 THEN
  triggered = true
END IF

Formato

📈 *Dia atipico — positivo!*
─────────────
Hoje: *R$ 8.000,00*
Media 30d: R$ 1.500,00
Desvio: 3,2σ (>2σ = notavel)
Por que 2σ
Numa distribuição normal, 2σ captura ~2.5% dos dias (top ou bottom). É o sweet spot: alerta só os dias realmente atípicos, sem barulho constante.

Automações de Agendamento (Mira cria → WhatsApp sai)

Quando a Mira cria um appointment (via confirmação de pendente no WhatsApp), o paciente deve receber automaticamente 3 mensagens ao longo do fluxo. Isso replica o que o engine JS do dashboard (js/agenda-automations.engine.js) faz quando um agendamento é criado pelo modal da Agenda — mas server-side, porque Mira insere via SQL e bypassa a camada JS.

As 3 mensagens

#QuandoTriggerTemplate
1Imediatamente após criaron_status com status=agendado"Confirmação Agendamento"
2D-1 às 9h (configurável)d_before com days:1, hour:9"Confirmação D-1"
3Dia da consulta às 8h (configurável)d_zero com hour:8"Chegou o Dia"
min_before NÃO vai pro paciente
O trigger min_before (30 min / 10 min antes) existe em wa_agenda_automations mas é alerta interno para a equipe. A Mira filtra isso explicitamente. Só dispara on_status + d_before + d_zero.

Onde a mensagem vem

Texto bruto mora em wa_agenda_automations.content_template (tabela já existente do dashboard). Suporta placeholders:

{{nome}}          → patient_name
{{data}}          → DD/MM/YYYY
{{hora}}          → HH:MM (5 chars)
{{profissional}}  → professional_profiles.display_name
{{procedimento}}  → procedure_name (default "Consulta")
{{clinica}}       → "Clinica Mirian de Paula"

Helper SQL _render_appt_template(template, appt record) faz o replace. A mensagem renderizada cai em wa_outbox com scheduled_at calculado pelo trigger (now() para confirmação, D-1 às 9h, D-zero às 8h). O Outbox Processor cron do n8n (x5f4MApEcSUX5hca, roda a cada 2min) pega mensagens com scheduled_at <= now() e envia via Evolution.

Por que replicar server-side?
O engine agenda-automations.engine.js roda no navegador — só dispara quando o dashboard está aberto criando appointments. Mira cria via SQL direto, então nunca acionaria o engine JS. A função wa_pro_fire_appointment_automations é a cópia SQL do mesmo comportamento, lendo a mesma tabela wa_agenda_automations, garantindo que a equipe continue editando templates pelo dashboard e as mudanças valem pros dois caminhos (dashboard ou Mira).

Fuzzy Search

Busca de pacientes usa pg_trgm com word_similarity (não similarity) — melhor pra casos onde a query é mais curta que o nome inteiro.

Algoritmo de score

score = GREATEST(
  word_similarity(query, name),
  0.95 if phone LIKE '%query_digits%',
  0.85 if name ILIKE '%query%'
)

WHERE (
  word_similarity(query, name) > 0.5
  OR name ILIKE '%query%'
  OR (query_digits != '' AND phone LIKE '%query_digits%')
)

Fluxo wa_pro_authenticate(phone):

  2. Busca em wa_numbers por right(8) match (compatível com "9 extra" brasileiro)

Como Adicionar uma Nova Intent

  1. Decidir se é Tier 1 (regex determinístico) ou deixar pro Tier 2
  2. Tier 1:
    1. Editar wa_pro_handle_message e adicionar ao CASE v_intent
    2. Editar wa_pro_execute_and_format e adicionar ao dispatcher
    3. Se precisar RPC nova, criar migration
    4. Adicionar área em INTENT_AREA se for permissão-sensível
  3. Tier 2:
    1. Adicionar tool em wa_pro_execute_tool (novo WHEN no CASE)
    2. Atualizar system prompt + tools array no workflow n8n (node "Tier 2 — Claude Haiku")
  4. Testes:
    1. Adicionar asserts em scripts/mira-tests.cjs
    2. Rodar node scripts/mira-tests.cjs
  5. Help dinâmico: atualizar a string em _fmt_help se for user-facing
  6. Commit + push: deploy automático via Easypanel
