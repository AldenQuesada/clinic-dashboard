Utilitários novos (js/utils/)

Money — matemática financeira em centavos

Sempre usar Money para qualquer aritmética monetária. Evita drift de float (0.1 + 0.2 !== 0.3).

// Soma de pagamentos valida contra total
var somaPag = Money.sum(pagamentos.map(p => p.valor))
if (!Money.eq(somaPag, total)) alert('Soma diferente do total')

// Divisão em parcelas arredondando certo
var valorParcela = Money.div(1000, 3)  // 333.33

// Parse de input BR ("1.234,56")
var valor = Money.parse(input.value)

// Formatação
Money.format(1234.56)        // "R$ 1.234,56"
Money.format(1234.56, false) // "1.234,56"

Testes: tests/money.test.js (32 casos). Nunca usar parseFloat(x).toFixed(2) em somas/comparações.

html — tag literal com escape por default

Substitui concatenação manual de strings HTML. Cada valor interpolado é escapado automaticamente — elimina a classe innerHTML + user data = XSS.

// ANTES (vetor XSS)
list.innerHTML = procs.map(p =>
  '<div>' + p.nome + '</div>'  // nome pode conter <script>
).join('')

// DEPOIS (seguro)
var H = window.html
list.innerHTML = procs.map(p => H`<div>${p.nome}</div>`).join('')

// Quando você PRECISA de HTML bruto (already-safe), use .raw()
list.innerHTML = H`<div>${H.raw(alreadyBuiltHtml)}</div>`

Testes: tests/html.test.js. raw() é opt-out explícito — nunca passar dado do usuário por raw().

Padrões e Gotchas

State alias mutation — nunca reatribuir

Quando um módulo consolida estado num objeto (_apptState) e mantém var _apptProcs = _apptState.procs como alias, qualquer reatribuição do alias quebra a sincronia.

// ❌ ERRADO — _apptProcs fica órfão, _apptState.procs continua apontando pro antigo
_apptProcs = []

// ✅ CERTO — muta in-place, alias continua válido
_apptProcs.length = 0

Foi um dos 4 bugs de regressão pós-Premium. Memória: feedback_state_alias_mutation.md. Aplicar o mesmo princípio para objetos (Object.keys(o).forEach(k => delete o[k])).

Event delegation vs onclick direto

Event delegation (data-action + listener único no container) é elegante mas sensível: se o listener não é instalado OU bubbling é bloqueado por propagação stop, o clique silenciosamente não funciona.

Regra prática:

  • Use delegation quando tem muitos botões dinâmicos (listas longas) e a interação é simples (click único)
  • Use onclick="fn(...)" direto para interações críticas em modais complexos — é determinístico e debug é trivial

Onclick direto exige expor a função em window — isso é catchado pelo typecheck via _globals.d.ts.

Cache-buster: sempre bumpar ao mudar JS

O Easypanel auto-deploya, mas o browser + Cloudflare podem servir versão antiga por horas se o cache-buster não mudar.

<!-- index.html -->
<script src="js/agenda-modal.js?v=20260411r" defer></script>

Sempre bumpar o ?v= (ex: r → s) quando mexer no JS que o HTML carrega. Já temos um script separado pra isso no futuro (item #9 do hardening).

RPC signature mismatch → NOTIFY pgrst

Quando você muda a assinatura de um RPC PostgreSQL (params adicionados/removidos), o PostgREST mantém cache de schema por até 10 minutos. Isso gera o erro famoso:

"Could not find the function public.xxx(...) in the schema cache"

Solução: notificar reload explícito via psql direto:

await c.query("NOTIFY pgrst, 'reload schema'")

Todos os scripts de apply de migration devem incluir esse NOTIFY no final.

Dirty watcher pattern (profModal)

O modal de profissional tem um array watchIds que dispara dirty marker para revelar o botão Salvar. Ao adicionar qualquer novo campo:

// js/professionals.js
const watchIds = ['sp_nome', 'sp_salario', 'sp_valor_consulta', /* ... */]

Se esquecer, o usuário digita e o botão Salvar nunca aparece. Memória: feedback_profmodal_dirty_watch.md.

Regressões pós-Premium (4 bugs) Corrigidas

Após o push do Premium round, o usuário reportou 3 bugs reais + 1 follow-up relacionado. Abaixo, cada um com causa raiz e fix.

R#1 — Procedimento não deletava nem selecionava cortesia

Causa: O item #6 do hardening trocou onclick="apptFoo()" por data-action + event delegation em #apptModal. A delegation teoricamente funcionava, mas na prática não disparava (provavelmente alguma propagação stop ou listener bind antes do DOM estar pronto).

Fix: Revert seletivo do _renderApptProcs e apptRenderPagamentos para onclick="..." direto, mantendo html\`\` tag pra escape de XSS. Commit e4acaa8.

Lição: Onclick direto é determinístico em árvores DOM complexas. Priorizar simplicidade quando o ganho da delegation é pequeno.

R#2 — _apptProcs órfão após troca de tipo

Causa: O item #5 do hardening consolidou estado em _apptState e manteve var _apptProcs = _apptState.procs como alias. Mas em 2 pontos do código havia _apptProcs = [] — atribuição que criava um array novo e deixava _apptState.procs apontando pro antigo.

Consequência: depois da troca de tipo, clicar nos procs não respondia porque o estado interno estava dessincronizado do rendering.

Fix: _apptProcs.length = 0 em ambos os pontos (_apptClearProcedimentoData + reset no openApptModal). Commit e4acaa8.

Lição: Consolidar estado via alias exige mutação in-place. Salvo na memória como feedback_state_alias_mutation.md.

R#3 — Valor da consulta "cruzando" pro procedimento

Causa: Na troca de tipo, o _apptHasConsultaData() retornava false quando o valor era 0 ou campos vazios, pulando o confirm() e o clear. Daí o appt_valor antigo permanecia.

Fix: Sempre zerar appt_valor + apptResetPagamentos() na troca de tipo, mesmo quando o confirm não dispara. Commit e4acaa8.

R#4 — Pagamento travado em R$ 200 (valor consulta) com procs somando R$ 1.500

Sintoma reportado: "alocado: R$ 200,00 / R$ 1.500,00 — falta R$ 1.300,00" em modo procedimento, sem o usuário ter feito split.

Causa: apptAutoValorConsulta (disparado na seleção do profissional) setava cegamente _apptPagamentos[0].valor = valor_consulta mesmo quando o tipo era procedimento. Depois, apptAddProc só atualizava a linha se valor === 0, então ficava travado nos 200.

Fix em 3 camadas:

  1. apptAutoValorConsulta — early return se tipo === 'procedimento'
  2. Nova função apptSyncPagamentoTotal() que força a 1ª linha de pagamento a refletir _apptValorTotalPagar() quando há exatamente 1 linha (respeita split manual)
  3. Chamadas adicionadas em apptAddProc, apptRemoveProc, apptProcUpdate (cortesia/valor), apptCalcDesconto

Commit 320d0f4.

7Pagamentos múltiplos com Pago/Aberto, parcelas e comentário Done

Por quê: O modelo antigo tinha 1 forma de pagamento + 1 valor (string). Falhas:
  • Não suportava pagamento misto (Pix + Cartão é comum)
  • Não distinguia pago vs aberto — relatórios financeiros não conseguiam mostrar contas a receber
  • Não capturava parcelas em crédito/parcelado
  • Não permitia comentário (ex: "cliente pagou entrada PIX, restante no débito")
Reformulação: array de pagamentos, cada um com forma + valor + status + parcelas + valorParcela + comentário.

Estrutura de dados

pagamentos: [
  {
    forma: 'pix' | 'dinheiro' | 'debito' | 'credito' | 'parcelado' | 'entrada_saldo' | 'boleto' | 'link' | 'convenio',
    valor: number,
    status: 'aberto' | 'pago',
    parcelas: int,        // só usado se credito ou parcelado
    valorParcela: number, // auto-calculado: valor / parcelas
    comentario: string,
  }
]

UI da linha

Card por linha de pagamento contendo:

  • Linha principal: [Forma ▾] [R$ valor] [✓ Pago / ○ Aberto] [×]
  • Parcelas (condicional): aparece quando forma é credito ou parcelado — input nº parcelas + input R$ valor parcela (auto-calculado)
  • Comentário: sempre presente, opcional

Helper de auto-cálculo

// Quando muda valor ou parcelas:
if (_apptFormaTemParcelas(p.forma) && p.parcelas > 0) {
  p.valorParcela = +(p.valor / p.parcelas).toFixed(2)
}

Validações no save

  • Forma obrigatória em cada linha
  • Se forma é crédito/parcelado, parcelas >= 1
  • Soma de todas as linhas == total esperado (consulta paga OU procedimentos não-cortesia com desconto)
  • Tolerância: 0,01 (centavo de arredondamento)

Derivações no save (compat retroativa)

formaPagamento: pagamentos.length === 1 ? pagamentos[0].forma : 'misto'
statusPagamento:
  todos pagos             → 'pago'
  algum pago algum aberto → 'parcial'
  todos abertos           → 'aberto'

Data Hygiene — patient fantasma (incidente 2026-04-12)

Durante auditoria da anamnese descobrimos um registro contaminado que tornava impossível testar o fluxo. Documentado aqui pra referência e pra embasar o refactor arquitetural proposto no fim da seção.

Sintoma

Causa provável

Colisão aleatória de UUID é matematicamente impossível (2^122 possibilidades). A explicação plausível: algum script de import ou migração criou leads passando id vindo de outra fonte (ex: export de CRM antigo) que por acaso bateu com um UUID já alocado em patients. -- 1. Remove anamnesis_requests fantasma (3 de teste)
DELETE FROM anamnesis_requests WHERE patient_id::text = '9d3a23f6-...';

-- 2. Remove patient duplicata (Adilso fantasma)
DELETE FROM patients WHERE id::text = '9d3a23f6-...';

-- 3. Lead da Emanuelle preservado intacto

Adilso real permanece em 57653fb9.... Quando Emanuelle virar paciente de verdade, o fluxo cria row nova em patients com seu próprio UUID (sem colisão).

Follow-up Refactor arquitetural patients.lead_id

O design atual de patients.id === leads.id (compartilhamento de namespace de UUID) é fundamentalmente frágil:

  • Qualquer import/seed/migration pode criar colisão silenciosa
  • O JOIN fica implícito (ON p.id = l.id) e mascara o contrato
  • Impossível ter 2 leads diferentes que viraram patient sem conflito
  • Propagação de erro é difícil de auditar (bug ficou invisível até um paciente reclamar do nome errado)

Proposta:

-- Nova coluna FK explícita
ALTER TABLE patients ADD COLUMN lead_id uuid REFERENCES leads(id);

-- patients.id passa a ser gerado independentemente (gen_random_uuid)
-- lead_id aponta pro lead original (nullable: paciente pode não ter vindo de lead)
-- UNIQUE (lead_id) pra evitar 2 patients pro mesmo lead

-- Join vira explícito em todos os lugares:
--   JOIN leads l ON l.id = p.lead_id

Impacto:

  • Elimina a classe inteira de bugs de mismatch de UUID
  • Permite histórico: um lead pode virar patient, virar lead de novo (walk-in), etc
  • Join fica óbvio quando auditado — p.lead_id é autoexplicativo
  • Migração: precisa backfill (set lead_id = id pros existentes) + atualizar todos os consumers

Custo estimado: ~1 dia de trabalho + testes extensivos. Recomendado antes da camada 2 (anamnese/consentimento) porque vários triggers e RPCs dessa camada dependem desse join.

CREATE FUNCTION update_updated_at_column_snake()
RETURNS trigger AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Bloco DO dinâmico: substitui TODOS os triggers em
-- tabelas anamnesis_* que usavam a fn antiga
DO $$
DECLARE r record;
BEGIN
  FOR r IN SELECT ... WHERE relname LIKE 'anamnesis_%'
                        AND proname='update_updated_at_column'
  LOOP
    EXECUTE format('DROP TRIGGER %I ON public.%I', r.tgname, r.relname);
    EXECUTE format('CREATE TRIGGER %I BEFORE UPDATE ON public.%I
                    FOR EACH ROW EXECUTE FUNCTION
                    update_updated_at_column_snake()', r.tgname, r.relname);
  END LOOP;
END $$;
