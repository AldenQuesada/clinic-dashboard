1Auto-detecção Novo / Retorno do paciente Done

Por quê: A secretária marcava manualmente Novo/Retorno e errava com frequência — especialmente quando o paciente tinha histórico mas vinha em uma nova categoria. Métricas de retenção ficavam infladas/deflacionadas. Regra do negócio: se o paciente já tem ≥1 atendimento finalizado, é Retorno; caso contrário, Novo. Isso é objetivo e auditável.

Como

Função apptDetectTipoPaciente(pacienteId) em js/agenda-modal.js que filtra o cache de appointments por pacienteId + status === 'finalizado' ignorando o appointment em edição (via appt_id). Acionada em 2 pontos:

  • selectApptPatient() — quando você seleciona um paciente no dropdown de busca
  • openApptModal() — quando o modal abre em edição (paciente já preenchido)

Variáveis

CamadaNomeTipoValores
UI (HTML)appt_tipo_pacienteselectnovo | retorno
JS (objeto appt)tipoPacientestring'novo' | 'retorno'
DB (appointments)—Salvo no JSON do appointment localStorage + Supabase

Validações

  • Detecção é determinística: appts.some(a => a.id !== editId && a.pacienteId === pacienteId && a.status === 'finalizado')
  • O editId é excluído pra não contar o próprio appointment como histórico
  • Default é 'novo' se nenhum critério bate

7Pagamentos múltiplos com Pago/Aberto, parcelas e comentário Done

Por quê: O modelo antigo tinha 1 forma de pagamento + 1 valor (string). Falhas:
  • Não suportava pagamento misto (Pix + Cartão é comum)
  • Não distinguia pago vs aberto — relatórios financeiros não conseguiam mostrar contas a receber
  • Não capturava parcelas em crédito/parcelado
  • Não permitia comentário (ex: "cliente pagou entrada PIX, restante no débito")
Reformulação: array de pagamentos, cada um com forma + valor + status + parcelas + valorParcela + comentário.

Estrutura de dados

pagamentos: [
  {
    forma: 'pix' | 'dinheiro' | 'debito' | 'credito' | 'parcelado' | 'entrada_saldo' | 'boleto' | 'link' | 'convenio',
    valor: number,
    status: 'aberto' | 'pago',
    parcelas: int,        // só usado se credito ou parcelado
    valorParcela: number, // auto-calculado: valor / parcelas
    comentario: string,
  }
]

UI da linha

Card por linha de pagamento contendo:

  • Linha principal: [Forma ▾] [R$ valor] [✓ Pago / ○ Aberto] [×]
  • Parcelas (condicional): aparece quando forma é credito ou parcelado — input nº parcelas + input R$ valor parcela (auto-calculado)
  • Comentário: sempre presente, opcional

Helper de auto-cálculo

// Quando muda valor ou parcelas:
if (_apptFormaTemParcelas(p.forma) && p.parcelas > 0) {
  p.valorParcela = +(p.valor / p.parcelas).toFixed(2)
}

Validações no save

  • Forma obrigatória em cada linha
  • Se forma é crédito/parcelado, parcelas >= 1
  • Soma de todas as linhas == total esperado (consulta paga OU procedimentos não-cortesia com desconto)
  • Tolerância: 0,01 (centavo de arredondamento)

Derivações no save (compat retroativa)

formaPagamento: pagamentos.length === 1 ? pagamentos[0].forma : 'misto'
statusPagamento:
  todos pagos             → 'pago'
  algum pago algum aberto → 'parcial'
  todos abertos           → 'aberto'

10Alerta de pagamento aberto na chegada à clínica Done

Por quê: O ponto natural pra cobrar é quando o paciente chega — antes ele está ocupado, depois ele já saiu. A secretária precisava de um lembrete automático nesse momento exato, visível no instante em que o status muda para "Na Clínica". Sem isso, contas em aberto passavam batido.

Como

Hook em apptTransition() em js/agenda-smart.js:

if (newStatus === 'na_clinica') _alertPagamentoAberto(appt)

Função _alertPagamentoAberto(appt):

  1. Lê appt.pagamentos[] e filtra os com status !== 'pago'
  2. Compatibilidade retroativa: appts antigos sem array usam statusPagamento === 'aberto' | 'pendente' | 'parcial'
  3. Soma o total em aberto
  4. Se > 0, dispara _showToast('Pagamento em aberto', msg, 'warning')
  5. Fallback: modal inline amarelo se _showToast não estiver disponível

11Modal de finalização inteligente Done

Por quê: Duas regras de negócio convergem aqui:
  • Consulta paga sem procedimento = paciente pagou só pela avaliação, secretária precisa cobrar antes de finalizar
  • Consulta paga + procedimento fechado = a consulta vira "cortesia" e é descontada do valor dos procedimentos (prática de mercado: a avaliação é grátis se você compra o tratamento)

Como

Estado novo: _finalAppt guarda o appointment carregado, usado para cálculos. Função _finConsultaAberta(appt) calcula quanto da consulta ainda está em aberto:

function _finConsultaAberta(appt) {
  if (appt.tipoConsulta !== 'avaliacao' || appt.tipoAvaliacao !== 'paga') return 0
  var pagamentos = appt.pagamentos || []
  if (pagamentos.length === 0) {
    return (appt.statusPagamento === 'pago') ? 0 : (appt.valor || 0)
  }
  return pagamentos
    .filter(p => p.status !== 'pago')
    .reduce((s, p) => s + p.valor, 0)
}

Em _finUpdateTotal():

  • Sem procs adicionados + consulta em aberto: finConsultaAlert aparece em amarelo no topo, e o campo finValor é preenchido com o valor da consulta
  • Com procs + consulta em aberto: totalFinal = totalProcs - consultaAberta, com detalhamento visual: "Total Procedimentos R$ X − Consulta R$ Y (cortesia ao fechar procedimento) = Total a cobrar R$ Z"

Auditoria → Hardening (15 itens)

Após o ciclo inicial de features, fizemos uma auditoria crítica e fechamos as 15 fraquezas identificadas. Estado antes → depois de cada uma:

#1 — Money em float

Antes: parseFloat(...).toFixed(2) em todo lugar; 0.1 + 0.2 !== 0.3 fazia validação de soma travar com diferença de R$ 0,01.
Depois: js/utils/money.js trabalha em centavos (int). APIs Money.eq, Money.sum, Money.sumEq, Money.div, Money.parse (formato BR), Money.format. Migrado em apptUpdatePagamentosTotal, validação do save, apptUpdatePagamento.

#2 — Test suite (vitest)

Antes: Zero testes. Refactor financeiro era apostar.
Depois: tests/money.test.js com 32 testes + tests/html.test.js com 9 testes. 41/41 passando. Cobre: drift de float, soma/sumEq, parse BR, formato, lógica de _apptValorTotalPagar com cortesia, derivação de statusPagamento, _finConsultaAberta.

#3 + #4 — HTML template helper + XSS escape consistente

Antes: 80+ linhas de '<div>' + p.nome + '</div>'. Escape ad-hoc com .replace(/</g, '&lt;') em alguns lugares e ausente em outros.
Depois: js/utils/html.js com tag literal html\`...\` que escapa por padrão. Opt-out explícito via html.raw(). Migrado em apptRenderPagamentos, apptRerenderPagamentoRow, _renderApptProcs.

#5 — Estado global consolidado

Antes: _apptProcs, _apptPagamentos, _finalAppt, _multiProcChoice espalhados como vars módulo.
Depois: _apptState = { procs, pagamentos, multiProcChoice } com _apptStateReset() coordenado. Refs legacy _apptProcs/_apptPagamentos mantidas como aliases pros mesmos arrays (mutação in-place preserva closures).

#6 — Event delegation pattern

Antes: onclick="apptFoo(...)" em strings inline → exigia window.apptFoo exposto pra cada handler. Já caímos uma vez (apptOnProfChange não exposto).
Depois: data-action, data-idx, data-field nos elementos. Único listener delegado em _apptHandleDelegated ouve click/input/change no modal e dispatcha. Não precisa mais expor handlers no window.

#7 — Render seletivo (preserva foco)

Antes: Trocar forma de pagamento de uma linha rebuildava todas as linhas via innerHTML — inputs com foco perdiam.
Depois: apptRerenderPagamentoRow(idx) reescreve apenas [data-pagamento-row="idx"]. Outros inputs (comentário, valor) com foco continuam intactos durante a edição.

#8 — Single source of truth (saveOneAwait)

Antes: syncOne() fire-and-forget — UI mostrava sucesso mesmo se Supabase falhasse. Bug recorrente (memória feedback_finalize_must_sync).
Depois: AppointmentsService.syncOneAwait(appt) retorna { ok, error?, queued? }. Chamadores podem await e mostrar feedback real ou enfileirar offline.

#9 — Schema audit script

Antes: Drift entre tabela ↔ RPC ↔ frontend repo só era detectado quando o usuário reportava bug.
Depois: node scripts/audit-schema.cjs conecta no Supabase, lê information_schema + pg_proc, parseia o repo JS e reporta diffs. Já detectou bug equivalente em clinic_injetaveis (6 params errados, 14 omitidos) — registrado como follow-up.

#10 — Cortesia em relatórios financeiros

Antes: Cortesias por procedimento ficavam invisíveis. Cashflow, Mira, DRE não as reportavam.

Premium upgrades (7 itens)

Após o hardening, fizemos uma rodada Premium para elevar qualidade ao nível de aplicação production-grade. Foco em type safety, observabilidade, acessibilidade e visibilidade operacional.

P#1 — JSDoc + tsc --checkJs

O que: tsconfig.checkjs.json com checkJs: true + JSDoc nos utils + arquivo js/utils/_globals.d.ts declarando tipos para window.Money, window.html, window.Modal, window.Logger, window.FeatureFlags.
Por quê: Sem type system, erros como "função não exposta no window" só aparecem em runtime na produção (já caímos uma vez nesta sessão com apptOnProfChange). Agora npm run typecheck roda tsc --noEmit e captura tudo no CI.
Status: npm run typecheck passa limpo. Custo zero em runtime.

P#2 — Structured logger

O que: js/utils/logger.js com níveis (debug/info/warn/error), buffer ring (últimos 100 logs), persistência do nível em localStorage, hook onError(handler) opcional para envio remoto.
Por quê: console.warn espalhado pelo código não tem contexto, não tem nível, e não é inspecionável. Logger estruturado permite Logger.dump() no console pra ver os últimos 100 eventos com timestamp e contexto.
Uso: Logger.info('appointment:saved', { id, valor })

P#3 — Error boundary global

O que: js/utils/error-boundary.js com ErrorBoundary.install() que captura window.error e unhandledrejection. Mostra Modal.alert user-facing com throttle (5s), delega para Logger, e opcionalmente envia pra endpoint remoto.
Por quê: Erros JS hoje quebram silenciosamente — usuário vê tela travada sem feedback. Agora ele vê uma mensagem clara e a equipe técnica recebe o stack trace.
Inicialização: chamado no DOMContentLoaded em index.html:5816

P#4 — A11y no Modal manager

O que: Modal manager refatorado com:

  • role="dialog" + aria-modal="true" + aria-labelledby apontando pro título
  • aria-label="Fechar" no botão ×
  • Focus trap: Tab e Shift+Tab ciclam dentro do modal
  • Restore focus: ao fechar, devolve foco pro elemento que abriu o modal
  • Foco inicial automático no primeiro elemento focável

Por quê: WCAG AA compliance básica + UX teclado decente. Usuário com leitor de tela ou só teclado consegue usar todos os modais.

P#5 — Feature flag service

O que: js/utils/feature-flags.js com FeatureFlags.isEnabled(), enable(), disable(), list(). Defaults no código, override em localStorage (clinicai_ff_*).
Flags atuais: new_payment_block, cortesia_per_proc, audit_trail_ui, whatsapp_feedback_badge, error_boundary_remote, perf_lazy_modal_mount.
Por quê: Permite rollout gradual de features de risco. Você pode habilitar uma feature pra um único usuário (DevTools localStorage.setItem('clinicai_ff_xyz', '1')) antes de defaultar pra todos.

P#6 — Performance budget audit

O que: scripts/perf-budget.cjs que parseia index.html, mede tamanho de cada <script src>, classifica em verde (<50KB) / amarelo (<150KB) / vermelho (≥150KB), reporta top 15 e total.
Comando: npm run perf:budget
Achados imediatos:

  • 3.68 MB de JS no boot, 163 scripts
  • VERMELHO injetaveis.js (235 KB)
  • AMARELO 18 arquivos (lead-modal 142KB, api 132KB, agenda-smart 121KB, tags 114KB, cashflow.ui 106KB, procedimentos 100KB, ...)

Otimização (lazy load + code split) registrada como follow-up.

P#7 — Expansão dos testes

O que: Novos test files:

  • tests/logger.test.js — 9 testes (API, buffer, níveis, onError handler)
  • tests/feature-flags.test.js — 9 testes (defaults, enable/disable/reset, list)

Total de testes na sessão: money (32) + html (9) + logger (9) + feature-flags (9) = 59 testes, todos passando.
