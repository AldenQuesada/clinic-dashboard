1 Visao geral

Hoje a clinica tem uma instancia WhatsApp ("Mih") que e a Lara — atende pacientes externos. Esta proposta cria uma segunda camada paralela: cada profissional conecta o proprio numero (ou um dedicado) e fala com um assistente interno que consulta dados do sistema e responde diretamente no WhatsApp.

O que muda na pratica

A Dra Mirian, ao inves de abrir o sistema, manda uma mensagem no WhatsApp pessoal:

"tenho horario amanha?" "quanto a Maria me deve?" "faturei essa semana?" "qual minha comissao do mes?"

E recebe a resposta em segundos, formatada, direta. Sem precisar abrir laptop, sem login, sem cliques.

2 Os 2 mundos separados

A separacao e fisica + logica + de seguranca. Numero diferente, instancia diferente, workflow diferente, bot diferente, RLS diferente.

🌎 Mundo Externo

Numero(s): Oficial(is) da clinica
Bot: Lara (vende, atende, agenda)
Quem fala: PACIENTE
Acesso: Apenas dados do paciente em questao
Status: Ja existe (instancia "Mih")

🔒 Mundo Interno

Numero(s): Particular(es) dos profissionais
Bot: Mira (consultas, lembretes, comandos)
Quem fala: PROFISSIONAL
Acesso: Total (filtrado por permissao)
Status: A criar (esta proposta)
Cross-talk impossivel: roteamento checa wa_number_id antes de chamar qualquer bot. Mira nunca responde a paciente. Lara nunca responde a profissional. Garantia logica + fisica.

Camada 1 — Numeros (extensao da tabela existente)

ALTER TABLE wa_numbers
  ADD COLUMN number_type text DEFAULT 'clinic_official'
    CHECK (number_type IN ('clinic_official','professional_private')),
  ADD COLUMN professional_id uuid REFERENCES professional_profiles(id),
  ADD COLUMN access_scope text DEFAULT 'own'
    CHECK (access_scope IN ('own','team','full'));

access_scope define o que cada numero pode ver:

  • own — so seus pacientes/agenda/financeiro
  • team — agenda da equipe inteira (gerente)
  • full — tudo da clinica (owner/admin)

Camada 2 — Roteamento (n8n)

Webhook recebe a mensagem → identifica wa_number_id → checa number_type:

  • clinic_official → segue fluxo Lara existente (zero mudanca)
  • professional_private → vai pro fluxo Mira novo

Camada 3 — Autenticacao

Quando uma mensagem chega pelo numero do profissional:

  1. Match telefone do remetente com professional_profiles.phone
  2. Se nao bate → mensagem ignorada + log (alerta de uso indevido)
  3. Se bate → carrega o professional_id e access_scope
  4. Cada chamada de RPC passa esse contexto

Camada 4 — Intent Parser (NLU)

Mira recebe texto → identifica intencao via Claude API com tool use:

patient_query agenda_query financial_query action_command help

Camada 5 — RPCs read-only wa_pro_*

Todas com prefixo padronizado e validacao de telefone como primeiro passo. Exemplo:

CREATE FUNCTION wa_pro_my_agenda(p_phone text, p_date date)
RETURNS jsonb AS $$
DECLARE v_prof_id uuid;
BEGIN
  -- 1. Autentica
  SELECT id INTO v_prof_id FROM professional_profiles
  WHERE phone = p_phone AND is_active = true;
  IF v_prof_id IS NULL THEN
    RETURN jsonb_build_object('ok', false, 'error', 'unauthorized');
  END IF;

  -- 2. Retorna dados (filtrados pelo professional_id)
  RETURN ...
END $$ SECURITY DEFINER;

Camada 6 — Formatador de resposta

Mira nao manda JSON cru — formata pro WhatsApp com markdown WhatsApp (*negrito*, separadores), emojis estrategicos, alinhamento de colunas.

Camada 7 — Audit + Rate limit

Toda query do profissional logada em wa_pro_audit_log. Rate limit configuravel (default: 30 queries/profissional/dia). Alerta no painel admin se profissional faz queries demais (potencial vazamento).

4 Seguranca

Cada risco tem uma mitigacao concreta.

RiscoMitigacao
Profissional vaza dados financeiros da clinica access_scope='own' por padrao — so ve o proprio. Owner explicitamente da 'full'
Alguem rouba o celular do profissional 2FA opcional via codigo enviado no email do profissional pra acoes sensiveis
Numero do profissional cair em maos erradas Toda RPC valida telefone E checa se profissional esta ativo. Owner pode "bloquear acesso WhatsApp" no painel sem mexer no Evolution
Lara responder a profissional por engano Roteamento checa wa_number_id antes de chamar Lara. Cross-talk impossivel
Mira responder a paciente por engano Mesma protecao — numero do profissional nunca cai no fluxo Lara
Custos LLM explodirem Rate limit por profissional + cache de respostas comuns + uso de Claude Haiku (mais barato) pra parser
Compliance LGPD: Dados sensiveis (CPF, valores totais) sao logados em audit. Owner pode auditar quem consultou o que e quando.

6 Plano sugerido (Bloco A + B)

Voce pediu prioridade Pacientes + Agenda + Financeiro. Isso bate com as Fases 2-3-4. Mas as Fases 0-1 sao pre-requisitos obrigatorios.

Bloco A — Fundacao (Fases 0+1) · ~2 dias

  • Schema: extender wa_numbers, criar wa_messages_pro, wa_pro_audit_log
  • Migration de seed: marcar a instancia Mih como clinic_official
  • UI no painel: tela "Numeros WhatsApp" pra cadastrar/segmentar
  • n8n: novo workflow "Mira router" com webhook proprio
  • Mira basica: responde "ola", lista comandos, autentica

Bloco B — Consultas core (Fases 2+3+4) · ~6 dias

  • Fase 2 (Pacientes): 4-5 RPCs wa_pro_patient_* + intents
  • Fase 3 (Agenda): 4-5 RPCs wa_pro_agenda_* + intents
  • Fase 4 (Financeiro): 4-5 RPCs wa_pro_finance_* + intents
  • Cada uma com seu prompt no Mira

Bloco C — Avancado (Fases 5-8)

Implementado depois, conforme demanda. Sao melhorias incrementais sobre uma base ja funcional.

Total Bloco A + B = ~8 dias uteis pra ter a Mira em producao com as 3 capacidades principais (pacientes / agenda / financeiro) funcionando para 1 profissional inicial (Dra Mirian).
