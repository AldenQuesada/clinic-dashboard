1. Origem do problema

O sistema foi construido do zero por uma unica entidade (Claude AI), sessao a sessao, ao longo de semanas. Nao houve "diferentes maos" ou heranca de codigo legado. A inconsistencia de tipos foi introduzida pelo mesmo agente que deveria ter mantido a consistencia desde o inicio.

O que aconteceu

Sessoes iniciais (marco 2026)

Criei as tabelas leads e patients com id text. Os valores inseridos eram UUIDs gerados por gen_random_uuid()::text — ou seja, UUIDs validos armazenados como string. O tipo uuid nativo do PostgreSQL teria sido o correto desde o primeiro CREATE TABLE.

Trigger _convert_lead_to_patient

Para gerar o patients.id a partir de um lead, usei md5('patient_lead_' || NEW.id) — um hash hexadecimal de 32 caracteres sem hifens. Funciona como identificador unico, mas nao e um uuid valido. Preguica tecnica.

Sessoes seguintes (abril 2026)

Criei appointments e medical_records com patient_id uuid — ja usando o tipo correto. Neste ponto eu ja sabia que uuid era o tipo certo. Mas nao voltei para corrigir patients.id e leads.id.

FKs e RPCs acumulam band-aids

Tabelas como anamnesis_requests, cashflow_entries, patient_complaints precisavam referenciar patients.id. Como ele era text, as FKs foram criadas como text — decisao correta dado o erro na raiz. As RPCs que juntavam appointments.patient_id (uuid) com leads.id (text) passaram a usar ::text como compensacao.

Abril 2026 — auditoria revela

O usuario perguntou: "me preocupa que acabamos de fazer uma auditoria para padronizar e ainda temos Tipos inconsistentes". A auditoria profunda revelou o tamanho real do debito.

Principio violado feedback_code_quality_from_start.md: "NUNCA entregar codigo que precise refatorar depois — best practices desde o primeiro commit." Este relatorio documenta a violacao e a correcao.

4. Bombas descobertas na auditoria

BOMBA 1 — Trigger _convert_lead_to_patient usa md5() como ID 
v_patient_id := md5('patient_lead_' || NEW.id);
INSERT INTO patients (id, ...) VALUES (v_patient_id, ...);

md5() retorna abc123def456789... — 32 chars hex sem hifens. Nao e uuid valido. Se patients.id virar uuid, este INSERT falha com invalid input syntax for type uuid.

Os 2 pacientes atuais no banco tem UUIDs validos — logo este trigger nunca executou com sucesso, ou os pacientes foram criados por outro caminho.

BOMBA 2 — leads.id tem 12 tabelas dependentes (nao 1)

A estimativa inicial era que so quiz_responses.lead_id dependia de leads.id. A auditoria profunda revelou 11 colunas text + 2 uuid em 13 tabelas diferentes. Converter leads.id → uuid exige converter todas em cascata — blast radius 3x maior que o previsto.

Tabelas afetadas: budgets, interactions, lead_pipeline_positions, leads_audit, phase_history, quiz_responses, rule_executions, tasks, wa_birthday_campaigns, wa_conversations, wa_outbox.

BOMBA 3 — 8 RPCs aceitam p_patient_id text na signature

Nao quebram imediatamente (coercao implicita do PostgreSQL cobre text → uuid), mas ficam frageis. Se alguem chamar com valor nao-uuid, o erro vem do cast implicito em vez de vir da validacao — debugar fica mais dificil.

BEGIN;

-- ============================================================
-- Passo 0: Reescrever trigger _convert_lead_to_patient
-- ============================================================
CREATE OR REPLACE FUNCTION public._convert_lead_to_patient()
RETURNS trigger LANGUAGE plpgsql SECURITY DEFINER
SET search_path TO 'public' AS $fn$
DECLARE
  v_patient_id uuid;
BEGIN
  IF NEW.status NOT IN ('paciente','attending','patient') THEN RETURN NEW; END IF;
  IF OLD.status = NEW.status THEN RETURN NEW; END IF;

  BEGIN
    v_patient_id := gen_random_uuid();

    INSERT INTO public.patients (
      id, "leadId", "tenantId", name, phone, email,
      status, "totalProcedures", "totalRevenue", "createdAt", "updatedAt"
    ) VALUES (
      v_patient_id, NEW.id, NEW.clinic_id::text,
      COALESCE(NEW.name,''), COALESCE(NEW.phone,''), NEW.email,
      'active', 0, 0, now(), now()
    )
    ON CONFLICT ("leadId") DO UPDATE SET
      name       = COALESCE(NULLIF(EXCLUDED.name,''),  patients.name),
      phone      = COALESCE(NULLIF(EXCLUDED.phone,''), patients.phone),
      email      = COALESCE(EXCLUDED.email, patients.email),
      status     = 'active',
      "updatedAt" = now(),
      deleted_at  = NULL;
  EXCEPTION WHEN others THEN
    RAISE WARNING '[_convert_lead_to_patient] Erro: %. Lead id=%', SQLERRM, NEW.id;
  END;

  RETURN NEW;
END;
$fn$;

9. Licoes aprendidas

Erros cometidos

  1. Tipo errado na raiz: id text em vez de id uuid nas tabelas fundacionais. Se tivesse sido uuid desde o inicio, nenhum dos 17 casts, nenhuma das 3 bombas, nenhuma hora desta auditoria teria sido necessaria.
  2. md5() como gerador de ID: Funciona como hash unico, mas nao e uuid valido. gen_random_uuid() ja existia no PostgreSQL e deveria ter sido usado desde o primeiro dia.
  3. Nao corrigir ao perceber: Quando criei appointments com patient_id uuid, eu ja sabia que o tipo correto era uuid. Naquele momento, o custo de voltar e corrigir patients.id era minimo (2 rows, 0 FKs naquela epoca). Escolhi seguir em frente. Esse "depois eu arrumo" custou 17 funcoes com band-aids, 3 bombas, e 2 sessoes de auditoria.
  4. Band-aids acumulativos: Cada ::text adicionado nas RPCs era um sinal de que algo estava errado na raiz. Em vez de corrigir a raiz, adicionei mais casts. Debito composto.

Regras para nao repetir

Regra 1: IDs sao sempre uuid nativo

Toda coluna que armazena um identificador unico universal DEVE usar o tipo uuid do PostgreSQL. Nunca text, nunca varchar. Sem excecoes.

Regra 2: gen_random_uuid() para gerar IDs

Nunca usar md5(), encode(), ou qualquer funcao que gere strings como ID. O PostgreSQL tem gen_random_uuid() nativo desde a versao 13.

Regra 3: Corrigir no momento, nao depois

Se ao criar a tabela N voce percebeu que a tabela 1 tem tipo errado, pare e corrija a tabela 1 ANTES de continuar. O custo cresce exponencialmente com cada dependencia adicionada.

Regra 4: Band-aid e sinal, nao solucao

Se voce esta adicionando ::text ou ::uuid para fazer um JOIN funcionar, algo esta errado na definicao de tipos. O cast compensa o problema — corrija a causa.

10.1 Obstaculos encontrados durante aplicacao

Obstáculo 1 — RLS policies bloquearam ALTER TYPE

O PostgreSQL impede ALTER COLUMN ... TYPE quando a coluna e referenciada por uma policy definition. O erro:

cannot alter type of a column used in a policy definition
Detail: policy anamnesis_requests_select on table anamnesis_requests depends on column "id"

Solucao: dropar as policies antes do ALTER e recriar depois com SQL identico. As policies nao armazenam dados — apenas definicoes de acesso.

Obstáculo 2 — 4 policies, nao 2

A primeira tentativa dropou 2 policies (anamnesis_requests_select e anamnesis_responses_select). Falhou novamente com:

policy anamnesis_answers_select on table anamnesis_answers depends on column "id"

Grep completo em pg_policy revelou 4 policies que referenciam patients.id via subqueries aninhadas:

  1. anamnesis_requests.anamnesis_requests_select — WHERE p.id = anamnesis_requests.patient_id
  2. anamnesis_responses.anamnesis_responses_select — via JOIN com requests → patients
  3. anamnesis_answers.anamnesis_answers_select — via JOIN responses → requests → patients
  4. anamnesis_response_flags.anamnesis_response_flags_select — via JOIN responses → requests → patients

Todas as 4 foram dropadas e recriadas na migration final.
