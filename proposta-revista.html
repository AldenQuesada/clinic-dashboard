01 · Fundamento

Visão geral do projeto

Revista digital mensal entregue via WhatsApp que educa pacientes e leads, reforça autoridade editorial da clínica, reativa base dormente e gera leads qualificados via quiz embutido — produzida sem editor, designer ou diagramador humano.

Três partes do sistema

01
Intake

Formulário onde você envia conteúdo bruto: brief, fotos, blocos de texto soltos, ordem desejada. Zero preocupação com layout.

02
Produção (Claude)

Claude lê o brief + playbook editorial, mapeia cada bloco para o template certo, escreve respeitando os contratos fixos, valida e insere no banco.

03
Leitor público

Página web horizontal scroll-snap, navegação por swipe/setas, gamificação com cashback, link assinado por HMAC para cada lead.

Objetivos de negócio

A
Autoridade

Posicionar a clínica como referência editorial em estética — não apenas prestadora de serviço.

B
Reativação

Trazer de volta pacientes dormentes (180+ dias) com conteúdo — não com oferta fria.

C
Conversão

Transformar leitura em agendamento via quiz personalizado + cashback contextual.

Princípios editoriais

  1. Naturalidade sobre transformação — textos nunca prometem "apagar anos". Sempre falam em "deixar aparecer quem você é".
  2. Autoridade sem jargão — linguagem acessível, mas com profundidade técnica. Zero promessas milagrosas.
  3. Casos reais sobre stock photos — fotos próprias da clínica sempre que possível. Antes/depois com TCLE obrigatório.
  4. Enxuto sobre denso — cada página horizontal comporta texto curto. Obriga clareza.
  5. Sem emojis editoriais — emojis só em CTAs/WhatsApp final. Texto editorial é tipográfico.
02 · Pivô de Arquitetura

Por que intake-based, não editor visual

A primeira tentativa foi um editor WYSIWYG (drag-drop, preview ao vivo). Funcionava — mas exigia que você lutasse com layout, proporções, limites. Pivotamos para um modelo onde você só descreve a intenção e fornece a matéria-prima; Claude faz o resto.

Separação radical

Você

  • Brief (tema, tom, objetivo)
  • Fotos com legenda livre
  • Texto cru (anotações, conversas, cases)
  • Ordem das seções
  • Aprovação final

Claude

  • Escolhe o formato certo de cada bloco
  • Escreve respeitando limites de chars/palavras
  • Mantém o tom editorial
  • Valida cada página antes de inserir
  • Mapeia fotos para slots por contexto

Por que isso destrava a operação

  • Tempo seu cai de 4h para 30min — você só faz brief + matéria-prima. O editorial pesa em quem sabe escrever sob restrição.
  • Consistência editorial garantida — validator no banco impede que uma edição saia com texto fora do contrato.
  • Layout nunca quebra — Claude só usa templates testados, com slots dimensionados em px fixos e variantes mobile.
  • Extensível — pode adicionar um 21º formato em qualquer momento (sequência: playbook → seed → CSS → renderer → validator → gallery → intake).
Insight do projeto: a melhor interface entre humano e IA não é uma UI rica — é um formulário simples que captura intenção. A IA faz o trabalho de tradução para o output formal.

05 · Editorial

Playbook — a fonte-da-verdade

O documento docs/magazine-editor-playbook.md é o contrato inegociável de cada uma das 20 seções. Toda sessão de produção começa com leitura integral dele. Sem exceção.

O que cada contrato define

  • Quando usar — caso de uso típico daquele formato
  • Slots obrigatórios — sem os quais não renderiza bem
  • Slots opcionais — refinos
  • Limites de caracteres e palavras — checados pelo validator no banco
  • Regras de foto — aspect ratio, resolução mínima, composição
  • Tom de voz por seção — entrevista é diferente de capa
  • Checagens automáticas — número mínimo de itens em arrays, formato de campos
  • Exemplo preenchido real — referência usável

Validator no banco

A função magazine_validate_section(template_slug, slots) roda no PostgreSQL antes de qualquer insert em magazine_pages. Retorna:

{
  "ok": false,
  "errors":   ["foto_hero", "titulo > 40 chars"],
  "warnings": ["lede curto (< 100 chars)"]
}

Errors bloqueiam. Warnings passam com aprovação explícita. Mesmo se Claude tentar inserir direto via SQL, passa pela mesma régua.

Regra permanente: Se algum pedido conflita com o contrato do playbook, prevalece o playbook. Claude deve pedir ajuste explícito antes de violar.

08 · Fluxo

Intake do usuário

Página magazine-intake.html — 5 blocos sequenciais. Tempo esperado: 15 a 30 minutos por edição.

1
Brief da edição

Mês · Tema · Tom · Objetivo

Quatro campos curtos. Define o "Norte" da edição. Claude usa para escolher tom geral, ordem das chamadas e palavras-chave.

2
Biblioteca de fotos

Drag-drop · descrição livre por foto

Sobe pro Storage, registra em magazine_assets. Cada foto ganha uma ref (foto1, foto2…) usável nos campos de imagem das seções.

3
Menu de seções

20 cards categorizados

Capas · Estruturais · Matérias · Visuais · Interativos · Extras. Click adiciona ao sumário.

4
Sumário montado

Forms específicos por seção · validação live

Cada seção mostra só os campos do seu contrato. Contadores de char/palavra. Bolinha de status (verde/amarelo/vermelho) baseada no magazine_validate_section.

5
Referências

URLs, fontes, notas livres

Tudo que Claude deve consultar como fonte: estudos, posts de redes, links de produtos.

Salvar & Enviar

Botão "Salvar rascunho" grava em magazine_briefs sem submeter. Pode voltar depois pelo "Carregar rascunho".

Botão "Enviar pro Claude" valida tudo e muda status para submitted. Você me avisa: "rodou a edição de abril" — eu produzo as páginas em ~10-15 minutos.

09 · Fluxo

Produção pelo Claude

Sequência rigorosa que Claude segue ao consumir um brief submetido.

Inicialização

Leitura integral do playbook

Antes de qualquer ação, Read docs/magazine-editor-playbook.md. Regra gravada em memória persistente.

Brief + assets

Busca brief + fotos do banco

SELECT * FROM magazine_briefs WHERE id = ? + SELECT * FROM magazine_assets WHERE id = ANY(asset_ids)

Edição

Cria edição

magazine_create_edition(title, slug, theme, subtitle) — status draft.

Por seção

Editorialização + validação + insert

Para cada bloco do brief: extrai conteúdo cru, escreve respeitando contrato (limites, tom), monta objeto slots, chama magazine_validate_section. Se errors, ajusta. Se ok, magazine_add_page(edition_id, slug, slots, segment_scope).

Finalização

Marca brief done

UPDATE magazine_briefs SET status='done', processed_at=now(), edition_id=?. Devolve link de preview pro usuário.

Iteração

Ajustes pontuais

Usuário revisa em revista-live.html?edition=slug&preview=1 e pede mudanças específicas. Claude reescreve só o necessário via UPDATE magazine_pages.slots.

Aprovação

Publicação

magazine_publish(edition_id) — draft → published, published_at=now(). n8n dispatch (a implementar) entrega via WhatsApp segmentado.

13 · Tecnologia

Segurança HMAC

Links públicos da revista assinam o par (lead_id, edition_id) com HMAC-SHA256. Sem hash válido nenhuma RPC pública retorna dados em produção.

Como funciona

  1. Secret armazenado em magazine_config(key='hmac_secret') — 32 bytes random gerados na migration 06. Tabela com RLS bloqueada para anon.
  2. Função _mag_current_hmac_secret() SECURITY DEFINER — única forma de ler o secret.
  3. Função magazine_sign_lead_link(lead_id, edition_id) — gera o hash. Acesso só authenticated (server-side ou n8n).
  4. Função _mag_verify_lead_hash(lead_id, edition_id, hash) — valida em todas as RPCs do leitor.

Multi-tenant

Função _mag_current_clinic_id() lê auth.jwt() ->> 'clinic_id' com fallback para clínica única (single-tenant atual). Todas as RLS de tabelas tenant filtram por essa função.

RLS

  • magazine_templates — leitura pública (active = true), escrita só service_role
  • magazine_editions/pages/assets/reads/rewards/briefs — filtradas por clinic_id = _mag_current_clinic_id()
  • magazine_config — sem nenhuma policy pública (acesso só via funções SECURITY DEFINER)

Storage

Bucket magazine-assets com leitura pública (CDN otimizado), escrita apenas authenticated. Path convention: intake/{brief_id}/{uuid}.{ext} ou editions/{edition_id}/{uuid}.{ext}. Limite 10 MB.

15 · Operação

Roadmap imediato

Próximas duas iterações priorizam destrabar o primeiro envio em produção.

Sprint atual

Primeira edição publicada

Você submete o primeiro brief via intake → Claude produz 8-12 páginas → revisão → publicação. Validação end-to-end do fluxo intake-based.

Próximo sprint

n8n dispatch + tracking

Workflow que pega edições publicadas e dispara WhatsApp segmentado por RFM (D+0 todos, D+3 não-abriram, D+7 abriram-não-leram). Usa magazine_sign_lead_link para gerar URL personalizada.

Sprint seguinte

Painel de briefs em fila

Lista todos magazine_briefs com status submitted. Claude consome em ordem de chegada. Marca processed_at quando termina.

Médio prazo

Templates 21+

Conforme demanda surgir, criar novos formatos (timeline vertical, antes/depois com slider, vídeo embed, quiz inline). Sequência fixa: playbook → seed → CSS → renderer → validator → gallery → intake.

Queries úteis

-- Performance da edição:
SELECT
  e.title,
  COUNT(DISTINCT r.lead_id) FILTER (WHERE r.opened_at IS NOT NULL) AS opens,
  COUNT(DISTINCT r.lead_id) FILTER (WHERE r.completed) AS reads_80,
  COUNT(DISTINCT r.lead_id) FILTER (WHERE r.quiz_completed) AS quiz_done,
  SUM(rw.amount) AS total_cashback
FROM magazine_editions e
LEFT JOIN magazine_reads r ON r.edition_id = e.id
LEFT JOIN magazine_rewards rw ON rw.edition_id = e.id
WHERE e.slug = 'abril-2026-smooth-eyes'
GROUP BY e.id;
