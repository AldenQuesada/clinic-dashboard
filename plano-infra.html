0 Visão geral

Hoje: 1 container nginx servindo tudo em clinicai-dashboard.px1hdq.easypanel.host. Mexer em qualquer coisa reinicia o sistema inteiro.

Depois: 4 services no mesmo project Easypanel, cada um com deploy, logs e scaling independentes. Se o face-mapping pesar a CPU, escala só ele. Se mexer no CSS do dashboard, só o frontend reinicia.

Project: clinicai ├── frontend-dashboard [nginx:alpine] → dashboard.aldenquesada.site ├── facial-api [python fastapi] → interno (porta 8000) ├── backend-api [node/prisma] → api.aldenquesada.site └── outbox-worker [node, sem porta] → interno Já rodando em projects separados: alexa-bridge, n8n (não mexe) Supabase (externo, não muda)

Services do mesmo project se falam pelo nome DNS interno (ex: http://facial-api:8000). Sem sair pra internet, sem custo de banda, latência mínima.

! Antes de começar

Faça isso na sexta à noite, antes do fim de semana de migração

  1. Exportar env vars do service atual
    No Easypanel, abra o service clinicai-dashboard → aba Environment → copie tudo pra um arquivo local .env.backup.
    Abrir Easypanel
  2. Anotar URLs e certificados atuais
    Aba Domains de cada service — guardar lista num bloco de notas pra replicar igual depois.
  3. Verificar que o deploy automático por push está funcionando
    Commit bobo em README → push → olhar logs do Easypanel → confirmar que fez deploy sozinho.
  4. Horário da janela
    Sábado de manhã ou domingo à tarde são bons — clínica fechada, sem risco de perder mensagem de paciente no meio da migração.

  2. Criar novo service Botão + (topo direito) → App
    Nome: facial-api
  3. Configurar source Aba Source

  4. Env vars Aba Environment
    Copiar do .env.backup APENAS as vars que o FastAPI usa (chaves de modelo, Supabase se usar).
  5. Configurar porta Aba Domains
    Porta interna: 8000. Não adicione domínio público ainda — vai ser acessado só pelo frontend via rede interna.
  6. Deploy Botão Deploy
    Aguardar build (5-10 min na primeira vez — download dos modelos do rembg é pesado). Acompanhar logs.
  7. Health check
    Eu abro terminal no service frontend-dashboard (botão Terminal no Easypanel) e rodo curl http://facial-api:8000/docs. Se retornar HTML do FastAPI, está vivo.

No código (eu faço)

  1. Editar nginx.conf
    Trocar proxy_pass http://127.0.0.1:8000 por proxy_pass http://facial-api:8000 no location /api/. Commit + push.
  2. Validar
    Abrir face-mapping no dashboard, carregar uma foto, testar detecção. Se funcionar, etapa 2 concluída.
Se quebrar: reverter nginx.conf pro commit anterior. Easypanel auto-redeploy em ~2 min e volta a servir o API embedded no container monolítico.

ATENÇÃO — risco real: hoje o n8n já dispara msgs de wa_outbox a cada 2min. Se ligar um worker novo em paralelo sem desligar o n8n, paciente recebe cada msg 2x. Auditar fluxo ATUAL antes. Só fazer essa etapa se houver motivo claro (ex: quer migrar do n8n pra worker Node próprio).

Se decidir fazer:

  1. Auditar: quem drena wa_outbox hoje?
    Workflow n8n outbox-processor-workflow. Pausar esse antes de ligar worker novo.
  2. Criar worker Node
    Script que dorme 2min, consulta wa_outbox pending, dispara via Evolution API, marca sent. Eu escrevo.
  3. Service novo no Easypanel + → App → nome outbox-worker
    Sem porta exposta, sem domínio. Só restart policy always.
  4. Desligar o workflow n8n equivalente NO MESMO MOMENTO do deploy
    Zero sobreposição. Se der ruim, liga o n8n de volta e mata o worker.

7 Envio direto híbrido — msgs de agendamento

Complexidade
Média
Risco
Médio
Onde testar
LOCAL primeiro

Objetivo: msg de confirmação instantânea ao criar agendamento vai direto pra Evolution (≈300ms), sem esperar o processor de 2min. Lembretes futuros (D-1, 30min antes, boas-vindas do dia) continuam no outbox.

Testar no ambiente local antes de qualquer coisa. Subir frontend na porta 8080 apontando pro Supabase e Evolution reais, criar agendamento de teste pra número próprio, validar que só 1 msg chega e chega rápido. Só depois committar pra prod.

Modelo híbrido

IMEDIATA (confirmação ao clicar "criar agendamento"): frontend → Evolution API (direto) ↳ falhou? → INSERT wa_outbox (scheduled_at=now, fallback) FUTURA (D-1, boas-vindas dia, 30 min antes): frontend → INSERT wa_outbox (scheduled_at=futuro) → processor drena no horário certo OPT-OUT / BLACKLIST: sempre via outbox (validação centralizada)

Passos

  1. Criar função utilitária WhatsAppDirect.send()
    Módulo isolado em js/services/whatsapp-direct.service.js. Faz POST pra Evolution /message/sendText/Mih com timeout de 5s. Retorna {ok, messageId} ou {ok:false, error, fallbackToOutbox:true}.
  2. Identificar pontos de chamada imediata
    Em agenda-automations.engine.js, os templates com day=0 e delay_hours=0 são imediatos. Esses ficam no caminho direto. Os demais (day=-1, delay_hours>0) continuam outbox.
  3. Implementar fallback transacional
    Se direct falhar (timeout, 4xx, 5xx), enfileirar no outbox com scheduled_at = now() e o processor pega no próximo ciclo. Paciente recebe com delay de 2min mas nunca perde a msg.
  4. Replicar validações no client (só as imediatas)
    Checar blacklist (wa_phone_blacklist) via Supabase ANTES de enviar. Se bloqueado, nem tenta. Reusa a RPC de guard existente.
  5. Logar em wa_log_message
    Envio direto também grava em wa_log_message (mesma tabela do webhook), pra manter histórico unificado pra inbox e auditoria.
  6. Teste local com número próprio
    Servidor local na porta 8080 → criar agendamento teste pro seu WhatsApp → verificar: (1) msg chega em ≤1s, (2) aparece na inbox, (3) wa_outbox NÃO tem linha pra essa msg imediata, (4) D-1 entra no outbox normalmente.
  7. Teste de falha (simular Evolution offline)
    Trocar URL da Evolution pra uma inválida localmente, criar agendamento, confirmar que fallback cai no outbox, e que processor (n8n ou worker) pega em 2min.
  8. Deploy condicional (feature flag)
    Variável DIRECT_SEND_ENABLED em js/config/env.js. Começar false em prod. Ativar true só depois do teste na clínica com 1-2 agendamentos reais acompanhados.

Riscos específicos dessa etapa

RiscoMitigação
Evolution cai → msg imediata perdidaFallback pra outbox em qualquer erro/timeout
Duplicação: direto + outboxFlag direct_sent=true impede enfileiramento do template imediato
Rate limit em broadcast (vários agendamentos batch)Broadcast mantém 100% outbox, só agendamento individual vai direto
Blacklist não aplicadaChecar via RPC antes do POST direto; se falha de check, cai no outbox
CORS da Evolution bloqueando browserTestar local — já sabemos que evolution.aldenquesada.site está no CSP e /headers já libera. Se CORS quebrar, só via backend-api.
Gotcha crítico: o bug da duplicação tripla (appt_/tag_/lead_) precisa estar 100% resolvido antes dessa etapa. Senão, ao invés de 3 INSERTs no outbox, você tem 3 fetches paralelos pra Evolution e o UNIQUE INDEX não protege mais. Verificar logs da Evolution depois de commitar abcccf0 em prod.
